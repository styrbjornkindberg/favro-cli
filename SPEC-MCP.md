# SPEC-MCP-001: Favro MCP Server

## Objective

Add an MCP (Model Context Protocol) server to the favro-cli package so that Claude Desktop and other MCP clients can access the full CLI from sandboxed environments. The MCP server acts as a thin shell-execution bridge — it does not duplicate CLI logic. As the CLI grows, the MCP server grows automatically.

**Target users:** Claude Desktop users who have favro-cli installed and want to use it from Claude's conversation interface.

---

## Commands / Tools Exposed

### `favro_help`

Get the CLI help text for any command.

| Field | Value |
|-------|-------|
| Input | `{ command?: string }` — optional subcommand string, e.g. `"cards list"` |
| Execution | Runs `node dist/cli.js --help` or `node dist/cli.js <cmd...> --help` |
| Output | Help text as a single text content block |
| Timeout | 15 seconds |
| Error handling | Non-zero exit → return stderr as text (not `isError`; help failures are non-fatal) |

### `favro_run`

Execute any favro CLI command.

| Field | Value |
|-------|-------|
| Input | `{ command: string }` — full argument string after `favro`, e.g. `"cards list --board abc123 --json"` |
| Execution | Runs `node dist/cli.js <args...>` via `execFile` |
| Output | stdout as text; stderr appended under `--- stderr ---` separator if non-empty |
| Timeout | 60 seconds |
| Error handling | Non-zero exit code → `isError: true`, stderr/message as content |
| Guardrails | None — the CLI's own `--dry-run`, scope, and confirmation flags remain the user's responsibility |

## Code Style

- Single file: `src/mcp-server.ts`, target ~100 lines
- No imports from `src/cli.ts` or any command files — shell-out only
- Use `child_process.execFile` (not `exec`) — safer argument passing, no shell injection
- Resolve binary path via `path.resolve(__dirname, 'cli.js')` at runtime (works from `dist/`)
- TypeScript strict mode (matches existing `tsconfig.json`)
- `#!/usr/bin/env node` shebang as first line
- Top-level `await` wrapped in an IIFE or `main()` function to satisfy CommonJS module system
- No `console.log` — stderr only for startup messages (`console.error`)

## Boundaries

### Always do
- Shell out to the compiled `dist/cli.js` binary — never import CLI internals
- Return the CLI's full stdout/stderr verbatim
- Set `isError: true` on non-zero exit codes for `favro_run`

### Never do
- Add guardrails or command filtering in the MCP layer
- Import from `cli.ts` or any command module
- Use `exec` with a shell string (use `execFile` + args array)
- Auto-run mutations on behalf of users (the CLI's own `--yes` / `--dry-run` flags control this)
